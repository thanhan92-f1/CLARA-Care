# Release Process (Semver + GHCR + Auto Notes)

This project uses `release.yml` to standardize semver releases and image publishing.

## What this workflow does

1. **Manual semver bump** (`workflow_dispatch`): `patch` / `minor` / `major`.
2. Creates next git tag (`vX.Y.Z`) from existing tags.
3. Supports direct **tag push** (`v*.*.*`) as release trigger.
4. Builds and pushes GHCR images for `api`, `ml`, `web` with tags:
   - `sha-<shortsha>`
   - `vX.Y.Z`
   - `vX.Y`
   - `vX`
5. Generates image manifest JSON containing digests and metadata.
6. Creates GitHub Release with autogenerated notes and uploads manifest.
7. Ghi `Release Summary` vào workflow summary (tag, SHA, registry, manifest).
8. CD pipeline có thể deploy trực tiếp theo `image_tag` immutable (`vX.Y.Z` hoặc `sha-<shortsha>`).

## Required permissions

Workflow permissions:
- `contents: write` (create tags, create release)
- `packages: write` (push GHCR images)

## Manual release steps

1. Go to **Actions** -> **Release** workflow.
2. Click **Run workflow**.
3. Choose:
   - `bump`: `patch` / `minor` / `major`
   - `target_branch`: default `main`
4. Workflow will tag, build, push images, create release notes, upload manifest.

## Deploy after release

Sau khi release hoàn tất:

1. Mở workflow `CD`.
2. Nhập `image_tag` đúng với tag vừa release (ví dụ `v1.4.2`) hoặc SHA tag (`sha-a1b2c3d`).
3. Deploy staging và chạy smoke.
4. Manual approve production promote.

## Safety checks có sẵn

- Reject tag không đúng pattern `vX.Y.Z`.
- Reject nếu tag đã tồn tại.
- Gắn đầy đủ `commit_sha` vào image manifest để trace build origin.

## Manifest output

File uploaded to release assets:
- `image-manifest-vX.Y.Z.json`

Example shape:

```json
{
  "release_tag": "v1.4.2",
  "commit_sha": "a1b2c3d4e5f67890...",
  "registry_repo": "ghcr.io/project-clara-hbt/clara-care",
  "generated_at": "2026-03-30T14:00:00Z",
  "images": [
    {
      "service": "api",
      "image": "ghcr.io/project-clara-hbt/clara-care/clara-api:sha-a1b2c3d",
      "digest": "sha256:...",
      "release_tag": "v1.4.2",
      "commit_sha": "a1b2c3d4e5f67890...",
      "pushed_tags": ["sha-a1b2c3d", "v1.4.2", "v1.4", "v1"]
    }
  ]
}
```

## Notes

- The workflow assumes Dockerfiles are present at:
  - `services/api/Dockerfile`
  - `services/ml/Dockerfile`
  - `apps/web/Dockerfile`
- GHCR repository naming is derived from `${GITHUB_REPOSITORY}` lowercased.
- Re-running the same release tag is idempotent for release asset upload (`--clobber`).
- Promotion pipeline is managed separately at `.github/workflows/cd.yml` (staging -> smoke -> production).
